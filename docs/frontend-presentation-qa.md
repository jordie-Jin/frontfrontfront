# 프론트 발표 예상질문 30선 (답변 포함)

아래 문항은 현재 SENTINEL 프론트 진행 내역(.context.md, docs 반영 이력) 기준으로 정리한 발표 예상 질문/답변입니다.

## 1) 제품/UX 관점

1. **Q. 왜 관리자/사용자 화면을 완전히 분리하지 않고 같은 화면에서 전환하도록 설계했나요?**  
   A. 운영자가 사용자 관점 KPI와 협력사 상태를 빠르게 비교해야 해서, 페이지 이동 없이 컨텍스트만 전환되도록 설계했습니다. 실제로 화면 전환 수와 클릭 수를 줄여 운영 효율을 높였습니다.

2. **Q. 로딩/에러 UX에서 가장 신경 쓴 부분은 무엇인가요?**  
   A. 단순 스피너가 아니라 `AsyncState` 패턴과 재시도 액션, 폴백 배너를 함께 제공해 “지금 어떤 상태인지”를 명확히 보여주도록 했습니다.

3. **Q. 왜 모달 중심 편집 플로우를 사용했나요?**  
   A. 공지/Q&A 수정 시 목록-상세-수정 맥락이 자주 끊기는 문제가 있어, 같은 화면 내 수정/확인/닫기 흐름으로 전환해 작업 단계를 줄였습니다.

4. **Q. 회원가입에서 약관 동의를 먼저 강제한 이유는 무엇인가요?**  
   A. 제출 시점 에러를 줄이고, 사용자에게 법적 동의 포인트를 선제적으로 안내하기 위해 입력폼 활성화 이전에 필수 동의 게이트를 뒀습니다.

5. **Q. 정책/약관을 별도 페이지가 아니라 모달로 제공한 이유는 무엇인가요?**  
   A. 회원가입 맥락을 유지한 채 확인/복귀가 가능해야 해서 모달로 처리했습니다. 이탈을 줄이고 재입력 부담을 줄이는 데 목적이 있습니다.

6. **Q. 랜딩 페이지 개선의 핵심은 무엇이었나요?**  
   A. 클릭 유도 요소를 과도하게 두기보다 핵심 메시지를 단순화했고, hover/시각 피드백을 복구해 제품 이해 흐름을 자연스럽게 만들었습니다.

7. **Q. 발표에서 UX 개선 효과를 어떻게 설명할 수 있나요?**  
   A. docs의 정량 시뮬레이션 기준으로 화면 전환/클릭/복구 단계가 감소했고, 고빈도 운영 시나리오일수록 절감 폭이 커진다고 설명할 수 있습니다.

8. **Q. 관리자 계정에서 기업 추가 Confirm을 막은 이유는 무엇인가요?**  
   A. 관리자 역할에서 의도치 않은 사용자 액션이 실행되는 리스크를 줄이기 위해 권한 기반으로 버튼과 핸들러를 비활성화했습니다.

9. **Q. 대시보드 KPI 카드 6개 체계로 고정한 이유는 무엇인가요?**  
   A. 백엔드 summary 응답을 단일 소스로 맞추고, 사용자 인지 부하를 줄이기 위해 핵심 지표만 일관된 레이아웃으로 노출하도록 정리했습니다.

10. **Q. 사용자 관점에서 가장 크게 좋아진 지점 한 가지는?**  
    A. 실패 상황에서도 즉시 재시도/폴백이 가능해 “멈춘 화면”처럼 보이지 않게 만든 점입니다.

## 2) API/아키텍처 관점

11. **Q. API 정규화 작업의 목적은 무엇이었나요?**  
    A. 구 경로/신 경로 혼재를 제거해 유지보수 비용을 줄이고, posts/companies/ai/files/user-qna 호출 규칙을 일관되게 맞추기 위함이었습니다.

12. **Q. companyCode를 companyId로 통일한 이유는 무엇인가요?**  
    A. API/타입/문서에서 식별자 표현이 달라 생기는 매핑 실수를 줄이기 위해 단일 식별자로 정규화했습니다.

13. **Q. 관리자 API와 일반 사용자 API를 어떻게 분리했나요?**  
    A. `admin/*` 경로를 별도로 사용하고, UI에서도 관리자 전용 액션과 사용자 액션을 역할 기반으로 분기했습니다.

14. **Q. Q&A 응답 매핑에서 name/content를 강조한 이유는 무엇인가요?**  
    A. 백엔드 응답 필드와 프론트 표시 필드 간 불일치로 누락되던 데이터를 안정적으로 렌더링하기 위해 명시적으로 매핑했습니다.

15. **Q. Q&A userId 쿼리를 제거한 이유는 무엇인가요?**  
    A. 사용자 목록 조회 계약 변경에 맞춰 불필요 파라미터를 제거했고, 서버 기준의 권한/필터 로직을 신뢰하도록 정리했습니다.

16. **Q. 관리자 사용자 전환에서 ID 타입 불일치 이슈는 어떻게 해결했나요?**  
    A. 숫자/문자열 혼용 환경을 고려해 비교/전달 타입을 정규화하여 선택 누락과 잘못된 매핑을 방지했습니다.

17. **Q. 리포트 다운로드를 새 탭 오픈이 아닌 fetch+blob으로 바꾼 이유는?**  
    A. 브라우저 팝업 정책 영향을 줄이고 파일명 제어 및 완료 시점 UX를 일관되게 제공하기 위해서입니다.

18. **Q. 첨부파일 다운로드 302/프리사인 URL은 어떻게 대응했나요?**  
    A. 파일 URL 조회 후 실제 다운로드 URL을 받아 blob 다운로드를 수행하도록 처리해 리다이렉트/만료 URL 케이스를 흡수했습니다.

19. **Q. AI 리포트 생성에서 상태 폴링을 둔 이유는?**  
    A. 비동기 생성 작업 특성상 즉시 완료를 보장할 수 없어 `requestId` 기반 상태 조회 폴링으로 완료/실패를 명확히 관리했습니다.

20. **Q. 선다운로드 우선 처리(기존 보고서 즉시 다운로드)의 장점은?**  
    A. 이미 존재하는 리포트까지 재생성하지 않도록 해 대기시간과 서버 부하를 줄이고 사용자 체감 속도를 개선합니다.

## 3) 안정성/운영 관점

21. **Q. API 실패 시 목데이터 폴백을 넣은 이유는 무엇인가요?**  
    A. 데모/운영 중 일시 장애에서도 화면 맥락을 유지해 기능 설명과 사용자 흐름이 끊기지 않게 하기 위함입니다.

22. **Q. 폴백은 데이터 신뢰성을 해치지 않나요?**  
    A. 폴백 배너를 함께 노출해 실제 데이터가 아님을 명확히 구분하고, 오해를 줄이도록 설계했습니다.

23. **Q. 리포트 생성 중복 요청은 어떻게 막았나요?**  
    A. 완료된 연도/분기 조합은 재요청을 차단하고 버튼 상태를 제어해 중복 생성/중복 폴링을 방지했습니다.

24. **Q. 워치리스트 삭제 404는 왜 별도 메시지로 처리했나요?**  
    A. “서버 오류”가 아니라 “이미 없는 대상”인 도메인 상황이어서, 사용자 행동 수정이 가능한 안내 문구로 분기했습니다.

25. **Q. 협력사 상세 렌더 에러 방어는 어떤 식으로 했나요?**  
    A. 인사이트/지표 배열 형태를 가드하고 null-safe 매핑을 적용해 일부 필드 누락 시에도 화면이 깨지지 않게 했습니다.

26. **Q. 외부 건강도 0%/스케일 혼선은 어떻게 해결했나요?**  
    A. 서버 값 기준 표기 원칙으로 정리하고, 0~1 값/백분율 해석 로직을 명확히 정돈해 일관된 표시를 보장했습니다.

## 4) 협업/품질 관점

27. **Q. 이번 프론트 작업에서 기술 부채를 줄인 사례가 있나요?**  
    A. 미사용 서비스/타입/유틸과 mock 라우트를 제거하고, KPI/식별자/경로 규칙을 정리해 코드 복잡도를 낮췄습니다.

28. **Q. 문서화는 어떻게 관리했나요?**  
    A. API 정규화 결과, AI 리포트 다운로드 동작, UX 전략 문서를 `docs`에 남겨 코드와 운영 설명의 간극을 줄였습니다.

29. **Q. 현재 남아 있는 리스크/보류 이슈는 무엇인가요?**  
    A. 관리자 답변 등록 후 서버 재조회 응답에 replies가 누락되는 경우가 있어, 현재는 로컬 병합 로직에 의존하는 구간이 남아 있습니다.

30. **Q. 다음 스프린트에서 프론트 우선 과제는 무엇인가요?**  
    A. 관리자 답변 등록 후 재조회 데이터 일치성 검증 자동화와, 질문자/질문 목록 추가 요구사항 반영이 우선입니다.
